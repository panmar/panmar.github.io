Trust the Source, Luke

Published on 2023-11-24. Tagged with c++, security.

It is a common belief that the only documentation programmer should rely on is the source code. Some experienced programmers argue that you should only trust in the disassembly. However, I'd like to present a case where neither of these beliefs holds true.

Consider this kernel code:

int x;
void kernel_func(int index) {
    if (index < secret_bounds) {
        x = x ^ detector[secret[index] * SOME_BIG_NUMBER];
    }
}

void kernel_func(int) PROC
        push    ebp
        mov     ebp, esp
        mov     eax, DWORD PTR _index$[ebp]
        cmp     eax, DWORD PTR int secret_bounds
        jge     SHORT $LN1@kernel_fun
        mov     ecx, DWORD PTR _index$[ebp]
        mov     edx, DWORD PTR int * secret[ecx*4]
        imul    edx, DWORD PTR int SOME_BIG_NUMBER
        mov     eax, DWORD PTR int x
        xor     eax, DWORD PTR int * detector[edx*4]
        mov     DWORD PTR int x, eax
$LN1@kernel_fun:
        pop     ebp
        ret     0

Could we get the value of the secret[index] provided we have only access to index, detector[] and we know SOME_BIG_NUMBER value? According to the source code and disassembly we should not, right?

Right?

Welcome to hardware land! CPU hate doing nothing. If there is a cache miss in the branch condition index < secret_bounds, which would make CPU stall for a while, CPU can calculate x = x ^ detector[...] doing so called speculative execution. If the condition, fetched from the memory, was false, it would simply not commit speculative results. So what's the problem? The problem is due to speculative execution CPU would fetch detector[secret[index] * ...] into cache, even if we did not meet the branch condition!

So if we evicted whole cache, executed kernel_func making CPU do branch prediction (e.g. run it with correct index a few times before) and ran:

auto start = rdtsc();  // high-precision timer
auto tmp = detector['A' * SOME_BIG_NUMBER];
auto end = rdtsc();
auto diff = end - start;

we could guess if secret[index] * SOME_BIG_NUMBER was 'A' by simply measuring diff (small time means it was fetched into the cache by speculative execution). We could then repeat the whole process for 'B', 'C', 'D', etc., and voilà!

This vulnerability, called Spectre/Meltdown, has been described in more detail in "Reading privilaged memory with a side-channel" by Jann Horn.

Trust no one, Luke ;)
